这两个月里我做了一些之前的我觉得很诧异的事情：注册一个新域名 `charchar.dev` 并把 blog 迁移到那里。\
讲真，这对我的好处真的那么大么，为什么我要用一个新域名呢。从做出决定的一刻到现在我都在一直想着这些，或者说担心这这些

这绝不正常，也绝不会是一个正确的决定

## 域名

其实域名是我从小以来第一个觉得最酷而自己又却最触碰不到的概念。但截至 2025 年，`sourlemonjuice.net` 已经持续存在了一年多了，这一切都变得如此熟悉但我也一直保持着警惕。

为什么？\
因为相比 github pages 这种 Platform as a Service，自己的域名和服务器基础设施需要一个维护者和金钱。这引入了一系列的不确定性，虽然对于功能网站来说这再合适不过了，但对于内容网站一旦发生变化造成的后果也是灾难性的。在过去所建立的所有链接都将无法使用，所写过的内容除了 web archive 也将无人得知。\
即使我是个会把源代码公开到 github 上的人（有些人会在本地构建完后推到一个单独的公开分支上），但我相信没有多少人会真的去看的。除了自己以外再也用没人在乎的东西，可以被叫做遗物吧

对于我的时间线而言，`sourlemonjuice.net` 是我用来公开所有我自己需要的公开服务的地方。不考虑 URL 长度，和优雅，只有单纯的功能性和来自网络的粗旷。\
至少我是这么想的，比如 `connectivity.sourlemonjuice.net/generate_204` 这种东西的存在都是经过考量的

而另一个新域名的背景则更为复杂：`charchar.dev`\
我不知道如果作为读者的你看到会怎么想，但 char 是我用在拉丁字母的 nickname。我总是说着不想，但 SourLemonJuice 实在是太长太长，而且即使作为一个命名空间也不完全独特。但总的来说够用，名字嘛向来如此

但由此引发的问题是作为访问者的割裂，和维护上的多余复杂度。我想在不同的地方上写清我的名字也可以是 char，但这并不够，非最主要用途的名字注定只会被自己亲近的人知道，这对我的 blog 有什么意义呢。\
我又有什么把以 SourLemonJuice 为名义认识我的人放到 `charchar.dev` 上呢?\
但它又足够的短，足以支撑起类似 `ip.charchar.dev` 这样的需要经常手动输入的 URL。仔细来看我简直是在逐渐把自己绑死在这两个域名上，互相牵制，互相依赖

这永远都不应该发生

但有哪些地方做错了，又有哪些是值得保留的呢？

## 方便输入和名字贴脸（什

就保留 `ip.charchar.dev` 的角度来看 `charchar.dev` 其实是一个有用的域名，但唯独不适合长时间保存内容和当作 API 接口。\
所以我也想借着这篇帖子重新捋清自己的想法：

- 永远不要根据他人的抱怨偏离自己的内心！这点真的对我自己很重要
- 单个内容依赖的基础设施越少越好
- PaaS 永远比自托管好，除非必要
- 如果觉得做了什么比不做会造成可预见但可以挽回的长期危害，不要做。`blog.charchar.dev` 就是例子

或许很快就会把 blog 从 `blog.charchar.dev` 改到 `blog.sourlemonjuice.net` 吧...
